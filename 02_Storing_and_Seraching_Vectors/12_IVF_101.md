# IVF (Inverted File Index) — Notes

## 0. What IVF Fixes from Tree-Based Indexes

**Tree problems (KD-tree, Ball tree, VP-tree):**
- Partition using geometric rules (axis cuts, spheres, distance shells)
- Pruning depends on clean separation between regions
- In high dimensions, distances concentrate → pruning fails → visit everything
- Degrades to brute force past ~20-100 dims
- All three tree types hit the same wall: the curse of dimensionality

**IVF solution:**
> Don't partition geometrically. **Learn** the partition from the data using clustering. Accept approximation from the start. Search only the most relevant clusters.

---

## 1. The Core Idea

Instead of recursive geometric splits, IVF does this:

1. **Cluster all vectors** into K groups using k-means
2. **Store each vector** in its cluster's inverted list
3. **At query time**, find the closest cluster(s), search only those lists

That's it. The "index" is just: which vectors belong to which cluster.

**Analogy:** Instead of cutting a city into grid squares (trees), you identify the natural neighborhoods (clusters) and search only the relevant ones.

---

## 2. Structure

### Centroids

K-means produces K **centroids** — representative vectors, one per cluster.

```
Centroid 1: [0.2, 0.5, ..., 0.1]    (768-dim)
Centroid 2: [0.8, 0.1, ..., 0.3]
...
Centroid K: [0.4, 0.7, ..., 0.9]
```

### Inverted Lists

Each centroid owns a list of all vectors assigned to its cluster:

```
Centroid 1 → [vec_42, vec_107, vec_3, ...]
Centroid 2 → [vec_88, vec_201, vec_15, ...]
...
Centroid K → [vec_5, vec_999, vec_73, ...]
```

Every vector appears in exactly one list.

**Why "inverted"?** Borrowed from text search — an inverted index maps a term to the documents containing it. Here, a centroid maps to the vectors belonging to it.

---

## 3. Building the Index

### 3A. Train (Learn the Partition)

```python
function IVF_TRAIN(vectors, K):
    """
    vectors : all N data vectors
    K       : number of clusters (partitions)
    """
    centroids = K_MEANS(vectors, K)
    return centroids
```

K-means runs iteratively:
1. Initialize K centroids (random or k-means++)
2. Assign each vector to nearest centroid
3. Recompute centroids as mean of assigned vectors
4. Repeat until convergence

**This is the expensive step** — runs once, offline.

### 3B. Assign (Populate Inverted Lists)

```python
function IVF_BUILD(vectors, centroids):
    """
    vectors   : all N data vectors
    centroids : K centroids from training
    lists     : inverted lists, one per centroid
    """
    lists = [[] for _ in range(K)]

    for v in vectors:
        # find which centroid v is closest to
        nearest_c = argmin(dist(v, c) for c in centroids)
        lists[nearest_c].append(v)

    return lists
```

Each vector is assigned to exactly one cluster.

---

## 4. Query-Time Search

```python
function IVF_QUERY(q, k, nprobe, centroids, lists):
    """
    q         : query vector
    k         : number of nearest neighbors to return
    nprobe    : number of clusters to search (the key tuning knob)
    centroids : K cluster centroids
    lists     : inverted lists (one per centroid)

    candidates : all vectors from searched clusters
    """
    # Step 1: find the nprobe closest centroids to q
    centroid_dists = [(dist(q, c), i) for i, c in enumerate(centroids)]
    closest_centroids = top_nprobe_closest(centroid_dists, nprobe)

    # Step 2: search only those clusters
    candidates = []
    for c_idx in closest_centroids:
        for v in lists[c_idx]:
            d = dist(q, v)
            candidates.append((d, v))

    # Step 3: return top-k from candidates
    return top_k_closest(candidates, k)
```

**That's the whole search.** Compare with trees — no backtracking, no pruning rules, no triangle inequality. Just: find the right clusters, brute-force within them.

---

## 5. Why This Works in High Dimensions (When Trees Don't)

Trees fail because their pruning rules depend on geometric separation that vanishes in high dims.

**IVF sidesteps this entirely:**
- No pruning rules to break
- No geometric boundaries to evaluate
- Clusters are learned from actual data distribution
- Similar vectors naturally cluster together (that's what k-means optimizes for)

The search within each cluster is brute force — but over a tiny fraction of the data.

**Math:** If K = 1000 and nprobe = 10, you search ~1% of the data. That's the speedup.

---

## 6. The nprobe Parameter (The Only Knob That Matters at Query Time)

### What It Controls

`nprobe` = how many clusters to search.

```
nprobe = 1   → search only the closest cluster (fastest, worst recall)
nprobe = K   → search all clusters (brute force, perfect recall)
nprobe = 10  → search 10 closest clusters (the sweet spot)
```

### The Tradeoff

| nprobe | Speed | Recall | What happens |
|---|---|---|---|
| 1 | Fastest | Low | Miss neighbors in adjacent clusters |
| 5-20 | Fast | Good | Hit most true neighbors |
| K | Slowest | Perfect | You're doing brute force again |

### Why You Need nprobe > 1

Same boundary problem as trees (file 08, section 6):

```
        |
   q •  |  • true nearest neighbor
        |
  Cluster A | Cluster B
```

Query is near a cluster boundary. True nearest neighbor is in the adjacent cluster. With `nprobe = 1`, you miss it. With `nprobe = 5`, you almost certainly catch it.

**Key advantage over trees:** `nprobe` is tunable at query time without rebuilding the index. You can trade speed for recall per-query.

---

## 7. Choosing K (Number of Clusters)

### Too Few Clusters (K small)

- Each cluster has many vectors
- Search within cluster is slow
- But centroid comparison is cheap

### Too Many Clusters (K large)

- Each cluster has few vectors
- Search within cluster is fast
- But centroid comparison gets expensive
- Clusters may become too small / uneven

### Rules of Thumb

```
K ≈ sqrt(N)          for moderate datasets
K ≈ 4 * sqrt(N)      common in FAISS documentation
```

| N (dataset size) | K (clusters) |
|---|---|
| 100K | ~300–600 |
| 1M | ~1000–4000 |
| 100M | ~10K–40K |

## 8. The Boundary Problem (IVF's Main Weakness)

### Hard Assignment

Every vector belongs to exactly one cluster. But vector space is continuous — cluster boundaries are artificial.

**What goes wrong:**
- Query near a boundary
- True nearest neighbor assigned to a different cluster
- If that cluster isn't in your `nprobe` set, you miss it

This is the **exact same problem** as trees (file 08, section 6). IVF doesn't solve the boundary problem — it just makes it tunable via `nprobe`.

### Soft Assignment (Partial Fix)

Some systems assign each vector to **multiple** clusters (e.g., the 2 or 3 closest centroids).

**Pros:** Better recall near boundaries
**Cons:** Index size grows, redundant storage and computation

---

## 9. IVF Build Cost vs Query Cost

| Phase | Cost | When |
|---|---|---|
| K-means training | Expensive (iterative, multiple passes) | Once, offline |
| Vector assignment | O(N * K) distance computations | Once, after training |
| Query: centroid search | O(K) distance computations | Every query |
| Query: list search | O(nprobe * N/K) distance computations | Every query |

**Total query cost:** O(K + nprobe * N/K)

Compare with brute force: O(N). If K = 1000 and nprobe = 10, you compute ~1000 + 10 * N/1000 distances. For N = 1M, that's ~11K vs 1M — roughly 100x speedup.

---

## 10. IVF vs Trees vs Graphs

| Property | Trees (KD/Ball/VP) | IVF | HNSW |
|---|---|---|---|
| Partition method | Geometric rules | Learned (k-means) | Graph connectivity |
| Works in high dims | No (~20-100d) | **Yes** | **Yes** |
| Build cost | O(N log N) | O(N * K * iters) | O(N * log N * efC) |
| Query cost | O(log N) ideal, O(N) high-d | O(K + nprobe * N/K) | O(log N) typical |
| Memory | Low (tree nodes) | Low (centroids + lists) | High (graph edges) |
| Tunable at query time | Limited | **nprobe** | **efSearch** |
| Online insert | Some | Append to nearest list | Yes (with graph update) |
| Exact search | Yes (low-d) | No (approximate) | No (approximate) |

## 11. IVF in Practice (What FAISS Does)

IVF is the backbone of FAISS (`IndexIVFFlat`, `IndexIVFPQ`).

**`IndexIVFFlat`:** Exactly what this file describes — cluster, then brute-force within clusters.

**`IndexIVFPQ`:** IVF + product quantization — compress vectors within each cluster. This is where IVF gets combined with compression for billion-scale search. (PQ is a separate topic.)

**Typical FAISS workflow:**
1. Train centroids on a sample of data
2. Add all vectors (assigned to clusters)
3. Set `nprobe` at query time
4. Search

---

## Key Takeaways

1. **IVF = k-means clustering + search only relevant clusters**
2. **Learns partition from data** instead of geometric rules — that's why it works in high dims
3. **nprobe controls the speed/recall tradeoff** — tunable per query, no rebuild needed
4. **Boundary problem still exists** — nprobe > 1 is how you handle it
5. **Build is expensive** (k-means training), but **query is cheap** (search tiny fraction of data)
6. **Foundation for production systems** — FAISS, Milvus, and others build on IVF
7. **Combines naturally with compression** (PQ) for billion-scale search
